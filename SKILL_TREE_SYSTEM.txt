1. SYSTEM ROLE

The skill tree is the player’s method of deliberate specialization.

It transforms passive Practice into meaningful identity through choice.

The system must:

Reflect how the player has actually played
Offer meaningful decisions (no filler)
Remain readable and navigable at all scales
Integrate diegetically with the Enchiridion
2. TERMINOLOGY
Skill

A domain of player activity (Valor, Finesse, Arcana, Delving, Forging, Artificing, Hearth)

Node

A single unlockable element in the skill tree

Node Types

Trait

Minor passive modifier
Enhances existing behavior
Must feel noticeable, not trivial

Technique

Unlocks a new action or mechanic
Changes how the player interacts with the game

Mastery

Major system-level unlock
Defines playstyle direction

Keystone

Build-defining modifier
Often introduces tradeoffs or extreme specialization
3. NODE DISTRIBUTION

Each skill contributes nodes to the global tree.

Per skill:

10–14 Traits
3–5 Techniques
3–4 Masteries
2–3 Keystones

Total per skill: ~15–24 nodes

Estimated full system:

~130–180 total nodes
4. TREE STRUCTURE

The skill tree is a single connected graph.

It is:

NOT separate trees
NOT isolated skill menus
A unified progression web
Structure Model
Skills act as directional influences (poles)
Nodes are placed by thematic alignment
Cross-nodes exist between skills
Layers

Outer Layer (Tier 1)

Always visible
No Practice requirement
Must feel valuable on their own

Mid Layers (Tier 2–3)

Require Practice + connections
Represent specialization

Deep Layer

High investment
Includes Obfuscated and Hidden nodes
5. NODE UNLOCK RULES

Each node requires:

1 skill point (always)
Connection to at least one unlocked node

Higher tier nodes may also require:

Minimum Practice in one or more skills
Cross-Nodes
Require Practice in multiple skills
Represent combined playstyles
Must reflect real player behavior, not arbitrary gating
6. VISIBILITY STATES

Nodes exist in three states:

Locked

Visible
Requirements shown
Description shown

Obfuscated

Visible
Requirements partially hidden
Description hidden

Hidden

Not visible
Revealed through conditions

Usage:

Mostly Locked
Some Obfuscated
Very few Hidden
7. PROGRESSION INTEGRATION

The system operates alongside the two-layer progression model:

Practice
Passive accumulation through gameplay
Rank
Active investment via skill points
Interaction
Practice unlocks eligibility
Rank defines commitment
Skill points are universal (not tied to a specific skill)

The gap between Practice and Rank is meaningful and must remain visible.

8. PRESENTATION RULES

The skill tree is presented through the Enchiridion projection system.

Rules
Nodes are NOT screen-space UI
Nodes exist in projection or page space
Interaction is tied to rendered geometry
Node Display

Each node must include:

Icon
Title
Description
Requirements
Hover Behavior
Tooltip anchored near node
Not detached overlay UI
Must feel physically tied to the node
9. VIEW MODES

The tree must support:

Focused View (Default)
Shows:
Current node
Parents
Nearby children
Reduces clutter
Exploration View
Full tree visible
Supports pan + zoom
Path Highlighting
Hovering a node highlights its connections
10. LAYOUT SYSTEM

Node positions are NOT manually defined.

Requirements

A layout engine must:

Interpret the tree as a graph
Assign positions dynamically
Prevent overlap
Maintain readability
Rules
Rendering code MUST NOT define layout
Layout must be deterministic
Must scale with node count
11. DESIGN CONSTRAINTS

The system must NOT:

Use fixed/manual node coordinates
Display the entire tree without navigation
Contain filler nodes with no meaningful impact
Require all skills for baseline progression
Collapse into visual clutter at scale
12. DESIGN PRINCIPLES
Every Node Must Matter

Each node must answer:

“Is the player excited to unlock this?”

If not:

Redesign it
Or remove it
Avoid Flat Stat Increases

Do NOT rely on:

+5% damage
+10% speed

Instead:

Bundle stats into meaningful mechanics
Create behavior changes
Specialization Must Be Expressive
Builds should feel distinct
Players should recognize playstyle differences
Cross-Skill Design is Critical
Cross-nodes are the most memorable part of the system
They must reflect real gameplay patterns
13. INTEGRATION WITH ENCHIRIDION

The skill tree is not a menu.

It is:

A projection of knowledge
An extension of the book
A diegetic interface
Interaction Flow
Player opens the book
Player selects a skill
Tree projects outward
Player navigates and invests
14. FUTURE EXTENSIBILITY

The system must support:

Additional nodes per skill
New cross-skill combinations
Visual upgrades (3D nodes, animation)
Mod integration

The structure must not require redesign when expanded.

15. FINAL PRINCIPLE

The skill tree is not just progression.

It is:

Identity
Memory
Expression of how the player has played

If it does not communicate that, it has failed.
